# Security

SuccessFactors Instance Explorer runs entirely on a user's local machine: an Express server bound
to `127.0.0.1`, processing files the user explicitly drops into the UI. There
is no shared service, no managed deployment, no user account system. The
threat model is therefore narrow — primarily, "what happens if a user uploads
a malicious or malformed file?" — but it still matters, because the user's
SuccessFactors metadata typically includes role names, permission grants, and
business-rule logic that an organization considers sensitive.

## Reporting a vulnerability

Please **do not** open a public GitHub issue for security reports.

Email **siempreluca@gmail.com** with:

- A description of the issue and its impact.
- Steps to reproduce (a minimal repro is hugely appreciated).
- Affected versions / commit SHAs if known.
- Whether you'd like to be credited in the fix commit / release notes.

You can expect an acknowledgement within a few days. As a single-maintainer
project there is no formal SLA, but I'll prioritize anything with realistic
exploitability over feature work.

## What is in scope

- Path traversal or file-overwrite via uploaded filenames or zip entries.
- XML external entity (XXE) or entity-expansion DoS in the OData / data-model
  parsers.
- Decompression bombs / disk exhaustion via uploaded zips.
- XSS via tenant-supplied labels rendered in the UI.
- Anything that lets a request to the local server affect files outside the
  current project workspace, or escape `127.0.0.1` binding when default
  config is in use.

## What is out of scope

- Issues that require setting `SUCCESSFACTORS_INSTANCE_EXPLORER_HOST=0.0.0.0` and running on an
  untrusted network — the docs explicitly warn against this.
- Risks from a user uploading their own legitimate data (e.g. "the tool
  surfaced a permission grant my admin didn't realize was there"). That's the
  feature, not a bug.
- Vulnerabilities in third-party dependencies that aren't reachable from
  SuccessFactors Instance Explorer's code paths. (For dependencies that *are* reachable, please
  do report.)
- Reports generated by automated scanners without a working repro.

## What I will not do

- I will not pay bug bounties (this is a free, MIT-licensed personal project).
- I will not request CVEs unless the issue genuinely warrants one and a
  coordinated disclosure timeline makes sense.
